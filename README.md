F5 - Fixtures & Property Setters in Install
Explain fieldname collision risk: what happens if your Custom Field has the same
fieldname as a field added by a future Frappe update?
A fieldname collision occurs when a Custom Field created by a developer uses the same fieldname as a field later introduced in the core Frappe or ERPNext framework. During upgrades, this can lead to migration errors such as duplicate column exceptions, conflicting field definitions, or unpredictable UI behavior.

Explain patching order: if Patch 1 creates a Custom Field and Patch 2 reads it, why
must they be separate entries in patches.txt and never merged?
Frappe executes patches sequentially based on the order defined in patches.txt, and each executed patch is recorded in the Patch Log table. If Patch 1 creates a Custom Field and Patch 2 reads or updates that field, they must remain separate entries to preserve execution order and dependency. If they are merged later, systems that have already executed Patch 1 will skip the merged patch because it appears already applied in the Patch Log, causing Patch 2 logic to never run.

G1 - Safe Monkey Patch with Version Guard
Analysis:

this guard prevents the monkey patch from being applied multiple times, without it causing unexpected behavior such as duplicated URL prefixes or recursive calls, and making debugging difficult.

Why is isolating patches in monkey_patches.py better than scattering them in

Isolating patches in monkey_patches.py is better because it keeps all risky runtime modifications in one controlled documented place

What is the correct escalation path: try doc_events first - then
override_doctype_class - then override_whitelisted_methods - then monkey patch.
Why is this the order?
1. doc_events - Preferred first because it hooks into existing document events (validate, on_submit, etc.) without changing core logic.
2. override_doctype_class - Used when you need to extend or modify a DocType’s controller logic, but still within frappe’s supported extension mechanism.
3. override_whitelisted_methods - Replaces an API method. It is more intrusive because it changes how system APIs behave globally.
4. Monkey patching – Last resort because it modifies framework functions at runtime, is fragile, and may break after framework updates.

Task D - Rate limiting & abuse protection:
what are the real risks of allow_guest=True endpoints? List 3 specific attack vectors.
Endpoints that allow guest access can be abused because they do not require authentication. If not protected properly, attackers can exploit them.
1. Brute-force data enumeration: Attackers may repeatedly call the API with different inputs to extract sensitive data.
2. Denial of Service (DoS): Attackers can send thousands of requests per second, causing:
    • high CPU usage
    • database overload
    • application slowdown or crash
3. Automated scraping / data harvesting: Bots can continuously call the endpoint to collect large amounts of data, such as job status or internal records.
